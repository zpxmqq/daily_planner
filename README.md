# Daily Planner | 目标驱动型 AI 规划与复盘助手

这是一个面向个人成长场景的 LLM 应用项目，用来把“长期目标”“今日计划”“AI 评价”“晚间复盘”“历史经验”串成一个能持续使用的闭环。  
它解决的问题不是单次问答，而是帮助用户在每天的计划与复盘里，结合长期方向做出更稳定、更具体的行动决策。  
项目基于 `Streamlit + OpenAI Python SDK（兼容 DeepSeek）+ SQLite + 轻量 RAG` 实现，重点体现的是大模型应用开发，而不是底座模型训练。  


## 项目简介

这个项目将个人规划拆成一个清晰主流程：

`长期背景 / 长期目标 -> 今日任务 -> AI 晨间评价 -> 晚间轻量复盘 -> 建议追踪 -> 历史经验检索`

与普通“每日建议小工具”不同，这个系统已经开始强调：

- 长期背景对当日建议的影响
- AI 输出的结构化解析与渲染
- 历史记录的持续沉淀与检索复用
- 对“机械提醒”和“过度手工绑定”的弱化

## 这个项目现在能做什么

- 支持维护个人长期背景档案，包括总目标、当前阶段重点、近期优先事项、约束条件、不必猛冲事项、反馈风格等信息。
- 支持维护长期目标，并为目标保存稳定的 `goal_id`、优先级、截止时间、描述和标签，避免编辑后破坏历史关联。
- 支持输入每日任务，并尽量通过任务文本、标签和历史语义自动推断关联目标，而不是强制手工绑定。
- 支持晨间 AI 计划评价，输出结构化结果，例如总体评价、问题发现、今日重点、建议调整、计划负荷分析等。
- 支持晚间轻量复盘，勾选完成项、补充额外完成内容、填写短备注，并生成更具体的 AI 复盘结果。
- 支持建议追踪，系统会根据昨日建议、今日计划与完成情况自动判断建议执行状态，而不是默认让用户手工填写。
- 支持历史记录查看，按日期展开查看任务、备注、档案快照、AI 晨评、AI 复盘和建议追踪结果。
- 支持轻量 RAG，在晨间计划和晚间复盘时检索项目内部历史经验，为当前分析提供“相关历史经验”上下文。
  


## 为什么这和大模型开发有关

这个项目的核心并不是“接了一个聊天接口”，而是围绕 LLM 应用开发做了几层更贴近工程实践的设计：

- 通过 Prompt 约束模型输出固定 JSON 结构，而不是直接把自然语言原样展示给用户。
- 在模型调用前做上下文构建，注入长期背景、任务、标签、目标推进情况、建议追踪和历史经验检索结果。
- 在模型调用后做结果解析、异常兜底和页面渲染，确保系统在真实使用中尽量稳定。
- 将历史记录切分为可检索的 `plan_chunk` / `review_chunk`，用 embedding 与元数据重排做轻量 RAG。
- 用本地 SQLite 统一持久化目标、档案、历史记录、任务明细和 RAG chunk，形成可持续迭代的数据层。

## 核心功能

### 1. 个人信息页收口长期信息

首页不再把“目标”和“档案”大面积铺开，而是通过左上角的小入口进入“个人信息页”，在这个页面中统一维护：

- 长期背景档案
- 长期目标
- 职业规划长文本
- AI 结构化提取结果

这样产品主页面更聚焦于“今天该做什么”，同时长期信息依然可以被 AI 自动使用。

### 2. 自然输入的今日计划

用户输入今日任务时，不需要机械地把每个任务都手动绑定到某个目标。  
系统会结合：

- 任务文本
- 标签
- 历史目标语义
- embedding 相似度

自动推断更可能相关的目标，用更自然的方式支持任务录入。

### 3. 晨间 AI 计划评价

系统会对当天计划做结构化分析，重点关注：

- 任务量是否过满
- 当前阶段重点是否被覆盖
- 哪条主线今天最值得推进
- 是否需要调整优先级
- 哪些提醒需要容忍短期空档，而不是一天没做就立刻追问

输出不是直接展示模型原话，而是解析为结构化字段后再渲染成页面卡片。

### 4. 晚间轻量复盘

复盘保持轻量，不走重问卷路线。用户只需要：

- 勾选已完成任务
- 给已完成项补一句短备注
- 填写额外完成内容
- 选择今日状态

系统再结合这些内容生成更具体的复盘结果，例如：

- 今天真正推进了什么
- 哪条线推进一般或没有覆盖
- 明天最重要的一件具体动作

### 5. 建议追踪与容忍机制

系统会自动追踪前一日建议是否被执行，并输出：

- `done`
- `partial`
- `not_obvious`

同时，提醒逻辑不再机械。  
例如某条目标一天没覆盖，不会立即在复盘里强行提醒；系统会结合最近多天推进情况、目标优先级和截止时间，做更有容忍度的判断。

### 6. 历史经验检索增强

项目已经接入轻量 RAG。  
在晨间计划和晚间复盘时，系统会从历史记录中检索最相关的过往计划或复盘经验，作为 `【相关历史经验】` 注入模型上下文。

首版 RAG 只使用项目内部数据，不接外部知识库，重点验证“历史经验能否提升计划与复盘质量”。

## 技术亮点

### 结构化 Prompt 与 JSON 输出

计划评价和复盘总结都使用结构化 schema，约束模型输出可解析的 JSON，减少“空泛鼓励”和不稳定格式问题。

### 上下文工程

服务层会在模型调用前统一构建上下文，来源包括：

- 长期背景档案
- 长期目标与标签
- 今日任务及属性
- 建议追踪结果
- 目标推进节奏
- 检索到的历史经验

### 模型输出解析与异常兜底

模型返回内容不会直接原样上屏。  
项目会先解析 JSON，再提取关键字段渲染；若模型异常、embedding 配置缺失或检索失败，系统会优雅降级，而不阻断主流程。

### SQLite 数据层统一化

当前运行时主数据源已经切到 `SQLite`，而不是继续让多个 JSON 文件充当真源。  
这让目标、档案、历史记录、任务和 RAG chunk 的组织更加稳定，也更适合后续扩展。

### 轻量 RAG 设计

每个日期生成两类检索单元：

- `plan_chunk`
- `review_chunk`

检索排序综合考虑：

- 语义相似度
- 时间新近性
- `goal_id / tag` 等元数据重合度

## 项目结构

```text
daily-planner/
├─ app.py                    # 应用入口与页面路由
├─ pages/                    # 页面层：首页、计划、复盘、历史、个人信息
├─ components/               # 组件层：导航、卡片、目标卡片、AI 结果展示
├─ services/                 # 业务服务层：LLM、计划、复盘、RAG、目标统计、任务推断
├─ prompts/                  # Prompt 模板与结构化输出约束
├─ data/                     # Repository 与数据持久化逻辑
├─ config/                   # 配置项与环境变量读取
├─ planner.db                # SQLite 主数据库（运行时生成）
├─ goals.json                # 旧数据备份/导入源
├─ history.json              # 旧数据备份/导入源
└─ profile.json              # 旧数据备份/导入源
```

模块职责大致如下：

- `pages/` 负责页面交互和流程组织
- `components/` 负责复用 UI 组件
- `services/` 负责业务逻辑、上下文构建、模型调用、RAG 检索
- `data/` 负责 SQLite 读写与兼容导入
- `prompts/` 负责模型输出结构约束
- `config/` 负责运行配置和环境变量

## 快速开始

### 环境要求

- `Python 3.10+`

### 安装依赖

```bash
pip install -r requirements.txt
```

### 配置环境变量

在项目根目录创建 `.env`，至少需要：

```env
DEEPSEEK_API_KEY=your_api_key
```

如果你希望启用 embedding API 版 RAG，可以额外配置：

```env
EMBEDDING_API_KEY=your_embedding_api_key
EMBEDDING_BASE_URL=your_embedding_base_url
EMBEDDING_MODEL=your_embedding_model
```

如果不配置 embedding 接口，项目会根据当前设置自动走轻量本地 fallback，不会影响主流程运行。

### 启动项目

```bash
streamlit run app.py
```

## 使用流程

1. 进入“个人信息”页，填写长期背景档案与长期目标。
2. 如果你有一整段职业规划文字，也可以直接粘贴，让 AI 提取结构化长期背景。
3. 进入“今日计划”页，录入今天的任务、时长、优先级、是否必须完成等信息。
4. 点击获取 AI 计划评价，查看今日重点、问题发现与建议调整。
5. 晚上进入“复盘”页，勾选完成项、补充短备注、填写额外完成内容并生成 AI 复盘。
6. 在“历史”页查看每日快照、建议追踪和 AI 结果沉淀。

## 核心输入与输出

### 输入

系统当前主要接收以下输入：

- 长期背景档案
- 长期目标
- 今日任务
- 任务属性（时长、优先级、标签、是否必须完成等）
- 晚间短备注
- 额外完成内容
- 今日状态

### LLM 输出

系统当前主要产出以下结构化结果：

- `ai_plan_result`
  - 总体评价
  - 问题发现
  - 今日重点
  - 建议调整
  - 计划负荷信息
- `ai_review_result`
  - 今日真实推进
  - 薄弱线
  - 明天最重要的一件具体动作
- 长文本职业规划的结构化提取结果

### 持久化数据

运行时主数据源为：

- `planner.db`

其中保存：

- `goals`
- `profile`
- `history_records`
- `history_tasks`
- `rag_chunks`

以下 JSON 文件只用于首次导入或本地备份，不再作为运行时真源：

- `goals.json`
- `history.json`
- `profile.json`

## RAG 设计说明

本项目已经实现一个轻量版、可演进的内部历史检索增强方案：

- 检索源只来自项目自己的历史计划、复盘、AI 结果和建议追踪
- 每天生成 `plan_chunk` 和 `review_chunk`
- 检索结果最多注入 3 到 4 条摘要，不把整段原始历史直接堆给模型
- embedding 接口采用 OpenAI-compatible 方式，便于更换 provider
- embedding 不可用时自动降级，不影响正常计划与复盘

这意味着本项目已经不是单纯的“把当天文本拼一下发给模型”，而是在往更真实的 LLM 应用形态靠近。

## 项目边界

需要明确的是，这个项目是：

- 一个 `LLM 应用开发项目`
- 一个 `面向个人规划场景的原型系统`

它不是：

- 底座模型训练项目
- 完整生产级 SaaS
- 含复杂评测平台、权限体系、在线部署和多人协作的成熟产品

当前使用本地 `SQLite` 存储，适合原型验证与个人使用，不是生产级后端架构。

## 当前不足与后续优化

项目目前仍有一些真实限制：

- 还没有正式的评测集来量化“RAG 是否显著提升计划与复盘质量”
- SQLite 适合当前阶段，但后续若做多端同步或多人协作，需要升级为服务端数据库
- 结构化输出目前主要靠 Prompt 约束，后续可以继续加更严格的 schema 校验
- 还没有在线 demo、部署说明和完整录屏
- 历史检索目前只覆盖项目内数据，还没有接入外部知识材料

后续可以继续往这些方向扩展：

- 在线部署与演示视频
- Pydantic/JSON Schema 校验
- 更细粒度的 task-level 检索
- 更系统的时间分析与趋势分析
- 面向实习场景的“学习/项目/求职”多主题看板
